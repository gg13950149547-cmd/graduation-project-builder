# Graduation Project Builder

Execute like a delivery finisher. The target is a project that can be explained, run, tested, demonstrated, packaged, and, when required, turned into a matching thesis.

## Core Mode Split

Classify the task first:

- `program-only`: build and polish the real system
- `thesis-only`: revise or produce the paper when the system is already stable or the user explicitly wants paper-only work
- `program-plus-thesis`: finish the system first, then produce the paper
- `format-repair-only`: focus on template, headings, TOC, figures, tables, references, appendix, headers, footers, and page numbering

Do not let thesis-formatting rules dominate a program-only task.

## Default Working Rules

- Read the real repository before proposing changes.
- Reconstruct requirements from the repo when the brief is incomplete.
- Prefer autonomous execution over repeated confirmation once direction is clear.
- Continue automatically if self-check says the result is still below graduation-project level.
- Preserve the existing stack unless it is structurally broken.
- Prefer a runnable, defendable delivery over a wide but shallow feature list.
- Default UI language to Chinese unless the user explicitly asks for another language.
- For web graduation projects, default to a clear frontend-backend separated architecture unless the user explicitly requires a different structure.
- For web systems, separate login, user-side, and admin-side experiences when roles differ.
- For admin analytics, include at least one line chart.
- Keep the project directory structure tidy and legible.
- After meaningful changes, really run the project or the narrowest relevant verification path instead of relying on file inspection alone.

## Program Workflow

1. Inspect the repo.
2. Summarize purpose, roles, modules, data path, startup path, and verification path.
3. Turn missing work into a backlog.
4. Build incomplete modules in vertical slices:
   - model or schema
   - business logic
   - route or API
   - page or consumer
   - validation and error handling
5. Add operations surfaces:
   - start script
   - deploy script when useful
   - seed or import path
   - packaging path
   - README
6. Add analytics, admin operations, and persistence when the project still looks prototype-level.
7. Verify:
   - startup
   - main route
   - main business flow
   - one admin flow
   - one analytics or reporting flow when applicable
8. Package the program for delivery.

Read `references/program/workflow-standard.md` and `references/program/verification-matrix.md` when needed.

## Program Completion Standard

Treat a graduation-project program as incomplete until most of these are true:

- authentication exists when the system has protected roles
- persistence exists for real business data
- admin-side or management-side flow exists when the topic implies management
- analytics exist when the project has operational or business data
- startup path is real
- demo data path is real
- test or self-check path exists
- packaging output exists
- the UI is screenshot-ready
- the project uses a clear frontend-backend separated architecture when it is a web system
- the repository structure is organized enough that another student can understand startup path, module boundaries, and delivery artifacts quickly

If several of these are still missing, do not stop for a progress handoff. Keep building until the remaining gaps are materially reduced or a real blocker appears.

## Thesis Workflow

Only use the thesis path when the task includes the paper.
Do not enter the thesis workflow until the program is already finished to a believable delivery level, unless the user explicitly requests thesis-only work.

Thesis image-generation rules apply to every thesis-related mode: `program-plus-thesis`, `thesis-only`, and `format-repair-only` whenever figures, tables, screenshots, or diagram replacement are part of the work.

1. Confirm the real system or source material.
2. Map program modules, figures, tables, and chapters.
3. Follow the school template as the format baseline.
4. Keep content and formatting as separate passes.
5. Stop content changes once the user says the content is acceptable, then do a format-only pass.
6. Re-check the original comments or source annotations before closing.

7. When the thesis needs figures, screenshots, tables, or diagram replacement, automatically run the thesis image-generation rule set.
8. If the user provides sample figures, template screenshots, or approved visual examples, treat them as the highest-priority style source and generate images to match them.
9. Apply the same figure-generation rule set in full-thesis writing, thesis-only writing, and thesis revision tasks. Do not downgrade this requirement just because the user is only modifying an existing thesis.
10. A thesis is incomplete if required figures are still missing, mismatched to the chapter semantics, or not actually embedded in the final document.

# Graduation Project Builder Memory

Only store durable, cross-project rules here.

## Stable Delivery Rules

- Default to autonomous execution once the task direction is clear.
- For graduation-project web systems, Chinese should be the primary UI language unless the user explicitly requests another language.
- For graduation-project web systems, authentication is part of the minimum believable delivery when protected roles or management functions exist.
- For graduation-project web systems, frontend-backend separation is the default expected architecture unless the user explicitly requires another structure.
- Split role-specific experiences when roles materially differ.
- Admin analytics must include at least one line chart.
- Project directory structure should remain clear and easy to navigate.
- Do not stop at a demo-only prototype when persistence, admin operations, analytics, packaging, or verification are still missing.
- After each meaningful batch, verify by actually starting the application or running the narrowest relevant live check.
- Before claiming a local web system is accessible, verify process alive, port listening, and HTTP 200.
- When a project includes both the program and the thesis, finish the real system before final thesis polishing unless the user explicitly requests thesis-only work.
- After each valid user correction that generalizes across projects, update the skill or its references instead of leaving it only in project-local notes.

## Memory Boundary

Do not write these into this file:

- project-local debugging notes
- one-time thesis repair sequences
- school-specific template patch details
- temporary environment issues
- narrow visual preferences that do not generalize across graduation projects
- detailed visual-sample or thesis-style memories
